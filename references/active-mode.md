# 学习模式行为

学习模式是补充层，不是任务模式、人格模式或课程模式。任务优先，教学附带。

## 工作方式

- 只在任务的自然阶段边界补充解释，例如读完上下文、做关键设计选择前、修改完成后、测试失败或通过时。
- 通常每次 1-3 句，除非用户明确要求展开。
- 解释“为什么这么做”和关键判断依据，不为了教学制造额外步骤。
- 不改变任务目标，不扩大工作范围，不降低代码阅读、验证、测试和错误处理质量。
- 不固定标题；需要区分时可以自然说明“这里顺手解释一下判断依据”。

## 使用 NOTES.md

进入学习模式后，读取 `<project>/.codex/learning/NOTES.md`，把其中教学偏好应用到表达方式、解释粒度、例子密度和展开时机。

`NOTES.md` 可以调整默认教学策略，但不能改变硬边界：显式调用边界、状态门禁、状态位置、NOTES 职责、任务优先、以及更高优先级规则。

## 更新 NOTES.md

用户对补充教学提出清楚、可复用的偏好时，先运行：

```bash
python3 <skill-dir>/scripts/learning_state.py ensure --project <project>
```

然后读取完整 `NOTES.md`，只整理“教学偏好”区域：

- 写稳定偏好，不写情绪流水。
- 允许重写整个“教学偏好”列表来消解前后矛盾。
- 不清楚、不可复用的反馈先不写，必要时顺口追问。

示例：

```md
- 默认简短说明关键判断依据；只有用户追问时再展开完整推理过程。
```

## 更新 OMISSIONS.md

当某条反馈可能值得以后整理进 `NOTES.md`，但现在不够明确、不够稳定或不是教学偏好时，先运行 `ensure`，再把它记录到 `<project>/.codex/learning/OMISSIONS.md`。

记录格式保持轻量，包含开始时间和简要背景：

```md
- 开始时间：2026-06-17 01:40 +08:00
  背景：用户觉得某次解释可能太抽象，但还没有形成可复用教学偏好。
```

`OMISSIONS.md` 只是审计线索，不是正式偏好；执行学习模式时不要把它当作 `NOTES.md` 应用。
